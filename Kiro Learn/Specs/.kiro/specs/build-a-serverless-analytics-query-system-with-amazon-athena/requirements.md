

# Requirements Document

## Introduction

This project guides learners through building a serverless analytics query system using Amazon Athena, Amazon S3, and AWS Glue. Learners will experience the full workflow of a modern cloud-based analytics pipeline: storing raw data in S3, cataloging it with AWS Glue, defining schemas, and running interactive SQL queries through Athena — all without provisioning or managing any infrastructure.

Understanding serverless analytics is a foundational skill for cloud practitioners. Traditional analytics platforms require significant operational overhead for cluster management, capacity planning, and infrastructure maintenance. Amazon Athena eliminates these burdens by allowing users to point directly at data in S3 and query it with standard SQL. This project teaches learners how to leverage that serverless model to go from raw data to actionable insights with minimal setup.

By completing this project, learners will understand how to organize data in S3 for analytics workloads, define and manage schema metadata through the AWS Glue Data Catalog, write and optimize SQL queries in Athena, control costs through data formatting and partitioning strategies, and secure access to their analytics resources. The project uses a realistic dataset so learners can practice queries that mirror real-world analytical scenarios.

## Glossary

- **data_catalog**: The AWS Glue metadata store that holds table definitions, schema information, and database references used by Athena to understand the structure of data in S3.
- **external_table**: A table definition in the data catalog that maps a schema to data files stored in S3, without moving or copying the data itself.
- **partitioning**: A data organization strategy that divides a dataset into segments based on column values (e.g., date or region), enabling Athena to scan only relevant subsets of data during query execution.
- **query_result_location**: A designated S3 path where Athena stores the output files generated by each executed query.
- **data_scanned**: The volume of data that Athena reads from S3 when executing a query, which directly determines query cost under the pay-per-query pricing model.
- **columnar_format**: A storage format such as Parquet or ORC that organizes data by column rather than by row, enabling Athena to read only the columns needed for a query and reducing data scanned.
- **workgroup**: An Athena resource that isolates query execution, tracks usage, and enforces configuration such as query result locations and data scan limits for a group of users or a specific purpose.
- **named_query**: A saved SQL statement in Athena that can be referenced and re-executed by name, enabling reuse of common analytical queries.

## Requirements

### Requirement 1: S3 Data Lake Storage Setup

**User Story:** As a serverless analytics learner, I want to organize raw data files in an S3 bucket structured for analytics queries, so that I understand how to prepare a data lake foundation for Athena.

#### Acceptance Criteria

1. WHEN the learner creates an S3 bucket for the analytics dataset, THE bucket SHALL have Block Public Access enabled to prevent unintended data exposure.
2. WHEN the learner uploads data files to S3, THE files SHALL be organized in a prefix structure that supports logical grouping (e.g., by dataset name or data type).
3. THE learner SHALL configure a separate S3 prefix or bucket as the query_result_location where Athena will write query output files.
4. IF the learner uploads data in a columnar_format such as Parquet, THEN THE files SHALL be readable by Athena and result in reduced data_scanned compared to equivalent CSV-formatted data.

### Requirement 2: AWS Glue Data Catalog Configuration

**User Story:** As a serverless analytics learner, I want to create a database and table definitions in the AWS Glue Data Catalog, so that Athena can understand the schema and location of my data in S3.

#### Acceptance Criteria

1. WHEN the learner creates a database in the AWS Glue Data Catalog, THE database SHALL appear as a selectable database within the Athena query editor.
2. WHEN the learner defines an external_table in the data_catalog with column names, data types, and an S3 location, THE table SHALL be queryable from Athena using standard SQL.
3. IF the learner defines a table schema that does not match the actual structure of the underlying S3 data files, THEN Athena queries against that table SHALL return errors or null values for mismatched columns.
4. WHEN the learner updates a table definition in the data_catalog (e.g., adding a column), THE updated schema SHALL be reflected in subsequent Athena queries without modifying the underlying data files.

### Requirement 3: Interactive SQL Querying with Athena

**User Story:** As a serverless analytics learner, I want to write and execute SQL queries against my S3 data using Athena, so that I can extract insights without managing any compute infrastructure.

#### Acceptance Criteria

1. WHEN the learner executes a SELECT query in Athena referencing a table defined in the data_catalog, THE query SHALL return results drawn from the corresponding S3 data files.
2. WHEN the learner runs queries using SQL features such as filtering (WHERE), aggregation (GROUP BY), sorting (ORDER BY), and joins, THE results SHALL correctly reflect the requested operations on the dataset.
3. WHEN a query completes execution, THE query results SHALL be automatically written to the configured query_result_location in S3 as a downloadable file.
4. WHEN the learner executes a query, Athena SHALL report the amount of data_scanned so the learner can understand the cost implications of the query.

### Requirement 4: Table Partitioning for Query Optimization

**User Story:** As a serverless analytics learner, I want to partition my data and table definitions by a meaningful column, so that I can observe how partitioning reduces the amount of data scanned and improves query cost efficiency.

#### Acceptance Criteria

1. WHEN the learner organizes S3 data files into a partition-compatible prefix structure (e.g., by date or category) and defines corresponding partition columns in the table definition, THE partitions SHALL be recognized by Athena after being loaded into the data_catalog.
2. WHEN the learner executes a query with a WHERE clause that filters on a partition column, Athena SHALL scan only the data in the matching partitions, resulting in less data_scanned compared to a query without partition filtering on an equivalent unpartitioned table.
3. IF the learner queries a partitioned table without filtering on the partition column, THEN Athena SHALL scan all partitions, and the data_scanned SHALL be comparable to querying an unpartitioned table.

### Requirement 5: Workgroup Configuration and Cost Control

**User Story:** As a serverless analytics learner, I want to create and configure an Athena workgroup with data scan limits, so that I understand how to manage and control analytics costs in a serverless environment.

#### Acceptance Criteria

1. WHEN the learner creates a workgroup in Athena, THE workgroup SHALL be configurable with a designated query_result_location and optional encryption settings for result files.
2. WHEN the learner sets a per-query data scan limit on a workgroup, THE workgroup SHALL cancel any query that exceeds the specified data_scanned threshold.
3. WHEN the learner executes queries within a workgroup, THE workgroup SHALL track and display metrics including total data_scanned and number of queries executed.
4. IF the learner switches between workgroups, THEN each workgroup SHALL maintain its own independent query history and configuration settings.

### Requirement 6: Saved Queries and Reusable Analytics

**User Story:** As a serverless analytics learner, I want to save frequently used SQL statements as named queries in Athena, so that I can understand how to build reusable analytics workflows.

#### Acceptance Criteria

1. WHEN the learner saves a SQL statement as a named_query with a descriptive name and optional description, THE named_query SHALL persist in Athena and be retrievable for future execution.
2. WHEN the learner executes a previously saved named_query, THE query SHALL run against the current data in S3 and return up-to-date results.
3. IF the learner creates multiple named queries within different workgroups, THEN each named_query SHALL be associated with and accessible from its respective workgroup.

### Requirement 7: Access Control for Analytics Resources

**User Story:** As a serverless analytics learner, I want to configure IAM policies that control access to Athena queries and the underlying S3 data, so that I understand how to secure a serverless analytics environment.

#### Acceptance Criteria

1. WHEN an IAM policy grants a user permission to execute Athena queries but restricts access to specific S3 data prefixes, THE user SHALL only be able to query data within the permitted S3 locations.
2. IF a user without Athena query execution permissions attempts to run a query, THEN THE service SHALL deny the request and the query SHALL not execute.
3. WHEN the learner configures server-side encryption on S3 data files using an S3-managed key or an AWS KMS-managed key, Athena SHALL be able to query the encrypted data provided the executing identity has the necessary decryption permissions.
