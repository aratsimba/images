

# Requirements Document

## Introduction

This project introduces the learner to event-driven architecture (EDA) by building a working system using Amazon EventBridge as the central event router. Event-driven architecture is a foundational pattern for modern cloud applications, where loosely-coupled components communicate through events rather than direct calls. By completing this project, the learner will understand how event producers and consumers operate independently, enabling scalable, resilient, and agile systems.

The learner will create a custom event bus, define rules with event patterns to filter and route events, configure multiple targets for event delivery, and implement event transformations. The project brings together several AWS serverless services—including EventBridge, AWS Lambda, Amazon SQS, and Amazon SNS—to demonstrate how real-world event-driven workflows are orchestrated. The learner will also work with EventBridge Pipes for point-to-point integrations and implement error handling strategies using dead-letter queues.

By the end of this project, the learner will have built a multi-component event-driven system that responds to both AWS service events and custom application events, routes them through rules and patterns, and delivers them to appropriate downstream targets for processing, notification, and archival.

## Glossary

- **event_bus**: A router in Amazon EventBridge that receives events and delivers them to zero or more targets based on matching rules.
- **event_pattern**: A JSON structure defined in an EventBridge rule that specifies which events the rule should match based on event attributes.
- **event_rule**: A configuration on an event bus that evaluates incoming events against a pattern and routes matching events to one or more specified targets.
- **event_source**: The origin of an event, which can be an AWS service, a custom application, or a third-party SaaS provider.
- **event_target**: The downstream AWS resource (such as a Lambda function, SQS queue, or SNS topic) that receives events matched by a rule.
- **dead_letter_queue**: An SQS queue configured to receive events that could not be successfully delivered to a target, enabling error handling and retry logic.
- **event_pipe**: An EventBridge Pipes resource that provides point-to-point integration from a single source to a single target, with optional filtering, enrichment, and transformation.
- **input_transformer**: An EventBridge feature that allows modification of event data before it is delivered to a target, selecting specific fields or restructuring the event payload.
- **schema**: A definition of the structure and format of an event, used to establish data contracts between event producers and consumers.
- **default_event_bus**: The event bus automatically provisioned in every AWS account that receives events generated by AWS services.

## Requirements

### Requirement 1: Custom Event Bus Creation

**User Story:** As a cloud architecture learner, I want to create a custom event bus in Amazon EventBridge, so that I can establish a dedicated router for my application's events separate from the default AWS event bus.

#### Acceptance Criteria

1. WHEN the learner creates a custom event bus with a unique name, THE event bus SHALL become active and ready to receive events.
2. THE custom event bus SHALL exist alongside the default event bus without interfering with default AWS service event routing.
3. IF the learner attempts to create an event bus with a name that already exists in the same account and region, THEN THE service SHALL return an error and the existing event bus SHALL remain unchanged.

### Requirement 2: Event Rule Definition with Pattern Matching

**User Story:** As a cloud architecture learner, I want to create EventBridge rules with event patterns that filter events by source, detail-type, and custom attributes, so that I understand how to selectively route events to specific targets.

#### Acceptance Criteria

1. WHEN the learner creates a rule with an event pattern matching a specific event source and detail-type, THE rule SHALL evaluate all incoming events on the associated event bus and match only those events that conform to the pattern.
2. WHEN multiple rules are defined on the same event bus, THE event bus SHALL evaluate each incoming event against all rules independently and deliver the event to the targets of every matching rule.
3. IF an incoming event does not match any rule's event pattern, THEN THE event bus SHALL not deliver that event to any target.
4. WHEN the learner defines a rule with a pattern that filters on nested fields within the event detail, THE rule SHALL correctly match events based on those nested attribute values.

### Requirement 3: Custom Event Publishing

**User Story:** As a cloud architecture learner, I want to publish custom events to an EventBridge event bus from my own application code, so that I understand how event producers emit events in an event-driven architecture.

#### Acceptance Criteria

1. WHEN the learner publishes a custom event with a specified source, detail-type, and JSON detail payload to a custom event bus, THE event SHALL be ingested by the event bus and evaluated against all rules on that bus.
2. WHEN the learner publishes multiple events in a single request, THE event bus SHALL process each event independently and route them according to their matching rules.
3. IF the learner publishes an event to an event bus that does not exist, THEN THE service SHALL return an error indicating the event bus was not found.

### Requirement 4: Multi-Target Event Routing

**User Story:** As a cloud architecture learner, I want to configure EventBridge rules to route events to multiple target types including Lambda functions, SQS queues, and SNS topics, so that I understand how a single event can trigger parallel processing, queuing, and notification workflows.

#### Acceptance Criteria

1. WHEN an event matches a rule that has a Lambda function as a target, THE event bus SHALL invoke the Lambda function with the event payload.
2. WHEN an event matches a rule that has an SQS queue as a target, THE event bus SHALL deliver the event as a message to the specified queue.
3. WHEN an event matches a rule that has an SNS topic as a target, THE event bus SHALL publish the event to the topic, which SHALL deliver it to all active subscriptions on that topic.
4. WHEN multiple targets are configured across one or more rules for the same event pattern, THE event bus SHALL deliver the matching event to all configured targets independently.

### Requirement 5: Event Transformation Before Delivery

**User Story:** As a cloud architecture learner, I want to use input transformers on EventBridge rules to reshape event data before it reaches a target, so that I understand how to decouple event structure from target processing requirements.

#### Acceptance Criteria

1. WHEN the learner configures an input transformer on a rule with an input paths map and an input template, THE event bus SHALL extract specified fields from the original event and deliver a restructured payload to the target.
2. WHEN an input transformer selects a subset of fields from the original event, THE target SHALL receive only the transformed payload and not the full original event.
3. IF a field referenced in the input paths map does not exist in the incoming event, THEN THE event bus SHALL deliver the transformed event with an empty or null value for that field.

### Requirement 6: AWS Service Event Monitoring

**User Story:** As a cloud architecture learner, I want to create rules on the default event bus that capture and respond to AWS service state-change events, so that I understand how EventBridge enables real-time monitoring and automated responses to infrastructure changes.

#### Acceptance Criteria

1. WHEN an AWS service emits a state-change event to the default event bus and a rule matches the event's source and detail-type, THE event bus SHALL deliver the event to the configured target.
2. WHEN the learner creates a rule matching a specific AWS service event pattern on the default event bus, THE rule SHALL only trigger for events matching that service and state change, not for unrelated AWS service events.
3. THE rule SHALL support filtering on specific attribute values within the event detail, enabling the learner to respond only to particular state transitions rather than all events from a given source.

### Requirement 7: Error Handling with Dead-Letter Queues

**User Story:** As a cloud architecture learner, I want to configure a dead-letter queue for an EventBridge rule target, so that I understand how to handle failed event deliveries and build resilient event-driven systems.

#### Acceptance Criteria

1. WHEN event delivery to a target fails after the configured retry attempts, THE event bus SHALL send the undelivered event to the dead-letter queue associated with that target.
2. THE dead-letter queue SHALL retain the original event payload along with metadata about the delivery failure, enabling the learner to inspect and troubleshoot failed events.
3. IF no dead-letter queue is configured for a target and delivery fails after retries, THEN THE event SHALL be dropped and the learner SHALL not be able to recover it.

### Requirement 8: Point-to-Point Integration with EventBridge Pipes

**User Story:** As a cloud architecture learner, I want to create an EventBridge Pipe that reads events from a source, optionally filters and transforms them, and delivers them to a target, so that I understand how Pipes complement event buses for point-to-point integration scenarios.

#### Acceptance Criteria

1. WHEN the learner creates a pipe with a specified source and target, THE pipe SHALL continuously read events from the source and deliver them to the target.
2. WHEN the learner configures a filtering pattern on the pipe, THE pipe SHALL deliver only events matching the filter criteria to the target and discard non-matching events.
3. WHEN the learner configures the pipe with an event bus as the target, THE pipe SHALL send events to the event bus, which SHALL then evaluate and route them according to its defined rules.
