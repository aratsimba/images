

# Requirements Document

## Introduction

This project guides learners through hosting a static website using Amazon S3, one of the most fundamental and practical skills in cloud computing. Amazon S3 static website hosting eliminates the need to provision and manage web servers, providing a cost-effective, highly durable, and scalable way to serve HTML, CSS, JavaScript, images, and other client-side content directly from an S3 bucket.

Static websites are ideal for personal portfolios, marketing pages, documentation sites, and single-page applications. By completing this project, learners will understand how to configure an S3 bucket for web hosting, manage public access settings, upload and organize website content, and handle custom error pages. These foundational skills apply broadly to serverless architectures and modern web deployment workflows on AWS.

The project focuses on the core S3 static website hosting workflow: creating and configuring a bucket, uploading website files, setting appropriate access permissions, and verifying the site is accessible via the S3 website endpoint. This is a learning exercise scoped to a single AWS service, and learners should clean up resources after completing the project to avoid ongoing charges.

## Glossary

- **static_website**: A website composed of fixed HTML, CSS, JavaScript, and media files that are delivered to the user's browser exactly as stored, with no server-side processing.
- **s3_bucket**: A container in Amazon S3 for storing objects (files and their metadata), identified by a globally unique name.
- **static_website_hosting**: An S3 bucket configuration feature that enables the bucket to serve its contents as a website, accessible via a region-specific website endpoint.
- **index_document**: The default HTML file (e.g., index.html) that S3 returns when a visitor requests the root URL or a directory of the website.
- **error_document**: An HTML file (e.g., 404.html) that S3 returns when a visitor requests a resource that does not exist on the website.
- **website_endpoint**: The region-specific URL automatically generated by S3 when static website hosting is enabled, used to access the hosted website via HTTP.
- **block_public_access**: A set of S3 security settings at the bucket or account level that prevent public access to bucket contents; these must be adjusted to allow public website hosting.
- **bucket_policy**: A JSON-based access policy attached to an S3 bucket that defines which principals can perform which actions on the bucket and its objects.

## Requirements

### Requirement 1: S3 Bucket Creation for Website Hosting

**User Story:** As a cloud learner, I want to create an S3 bucket in a chosen AWS Region, so that I have a storage container ready to hold my static website files.

#### Acceptance Criteria

1. WHEN the learner creates an S3 bucket with a globally unique name and a selected AWS Region, THE bucket SHALL be successfully created and visible in the S3 console.
2. IF the learner specifies a bucket name that already exists across any AWS account, THEN THE service SHALL return an error and no bucket SHALL be created.
3. THE bucket name SHALL conform to S3 naming rules, including lowercase letters, numbers, and hyphens only.

### Requirement 2: Static Website Hosting Configuration

**User Story:** As a cloud learner, I want to enable static website hosting on my S3 bucket and configure an index document and an error document, so that the bucket serves my content as a website with proper default and error pages.

#### Acceptance Criteria

1. WHEN the learner enables static website hosting on the bucket and specifies an index document, THE bucket SHALL be configured to serve the specified index document when visitors access the website endpoint root URL.
2. WHEN the learner specifies an error document in the static website hosting configuration, THE bucket SHALL serve that error document when a visitor requests a resource that does not exist.
3. WHEN static website hosting is enabled, THE S3 console SHALL display the region-specific website endpoint URL for the bucket.

### Requirement 3: Public Access and Bucket Policy Configuration

**User Story:** As a cloud learner, I want to configure my bucket's access settings so that website visitors can view the hosted content, so that I understand how S3 permissions control public access to objects.

#### Acceptance Criteria

1. WHEN the learner disables the Block Public Access settings on the bucket, THE bucket SHALL allow public access policies to be applied.
2. WHEN the learner attaches a bucket policy granting public read access to the bucket's objects, THE website content SHALL be accessible to unauthenticated visitors via the website endpoint.
3. IF Block Public Access settings remain enabled on the bucket, THEN THE bucket SHALL reject any bucket policy that grants public access, and website content SHALL not be publicly accessible.

### Requirement 4: Website Content Upload and Verification

**User Story:** As a cloud learner, I want to upload HTML, CSS, and image files to my S3 bucket and verify the website is accessible through the website endpoint, so that I can confirm my static website is live and functioning correctly.

#### Acceptance Criteria

1. WHEN the learner uploads an index document (e.g., index.html) and supporting files (CSS, JavaScript, images) to the bucket, THE objects SHALL be stored in the bucket and listed in the S3 console.
2. WHEN a visitor navigates to the bucket's website endpoint URL in a browser, THE S3 service SHALL return the index document content and render the website.
3. WHEN a visitor requests a path that does not match any object in the bucket, THE S3 service SHALL return the configured error document.
4. WHEN the learner uploads files organized in folder prefixes within the bucket, THE website SHALL serve those files using the corresponding URL paths relative to the website endpoint.
